# 性能证据与声明策略

性能取决于负载形态、MongoDB 拓扑、索引、网络延迟、缓存后端、文档大小、并发和硬件。monSQLize 因此不承诺固定加速倍数或通用开销百分比。

## 证据契约

任何数字性能声明在作为项目结论展示前，都必须记录以下证据：

| 字段 | 必需证据 |
|---|---|
| 负载 | 操作、选项、冷/热状态、并发和对照基线 |
| 环境 | Node、MongoDB/Driver、CPU、内存、操作系统和网络拓扑 |
| 数据集 | 文档数量、结构、索引和缓存填充状态 |
| 命令 | 其他维护者可执行的仓库命令或脚本 |
| 产物 | 原始样本、汇总统计、commit 和时间戳 |
| 预算 | 绑定该负载的回归阈值，而不是产品级通用保证 |

## 当前已验证范围

- `npm run test:performance` 是受控的 FunctionCache 回归守卫，验证 fixture 下的热路径/并发合并行为，不证明固定的生产加速倍数。
- `npm --prefix website run check:budgets` 约束构建资产和搜索索引的体积趋势；这是传输体积预算，不是页面加载时延声明。
- 批量写、查询缓存、populate、`findPage`、DataTask、SSH 和慢查询日志在建立可复跑负载与证据产物前只作定性描述。

选择批量大小、缓存策略、连接池和运维超时时，应对自己的真实负载做基准测试。
