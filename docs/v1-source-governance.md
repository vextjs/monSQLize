# v1-source 大测试文件治理策略

> 结论先行：**当前仓库对 `test/v1-source/**` 采取“vendored 原样保留 + 文档治理 + 渐进迁移”策略**，不做无收益的大规模机械拆分。

## 为什么不直接拆

1. `test/v1-source/**` 的首要职责是**保留 v1 行为事实**，不是追求当前仓库的最佳可读性。
2. 这些文件是兼容性基线，机械拆分很容易引入“搬运错误”或破坏历史断言分组。
3. 当前 file-size guard 只约束 `src/`，并不阻止 vendored 测试原样保存。

## 当前策略

| 区域 | 策略 |
|------|------|
| `test/v1-source/**` | 保持 vendored 原样，作为兼容事实来源 |
| 新增 TS / 当前主线回归 | 放入 `test/unit` / `test/integration` / `test/compatibility` |
| 热点兼容问题 | 先在当前主线补聚焦守卫，再决定是否从 `v1-source` 迁移 |

## 允许迁移的条件

满足以下任一条件时，才建议把 v1-source 中的断言拆到当前主线：

1. 某类回归在当前主线频繁出现，需要更快、更小粒度的聚焦测试。
2. 该断言已经不再依赖 v1 私有结构，迁移后能直接服务当前维护。
3. 迁移后能进入 `test:refactor-guard` / `test:unit` / `test:integration`，显著提升日常反馈速度。

## 禁止事项

- 不要为“让文件变短”而对 vendored 兼容基线做纯格式化拆分。
- 不要把 `v1-source` 当成新功能测试入口。
- 不要把当前主线 bug 只留在 v1-source 里验证，必须补当前主线的聚焦守卫。
